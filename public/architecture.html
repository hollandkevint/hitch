Draft and commit are separate paths. The write only fires through approval.

The whole safety story is one design choice, enforced in code: the thing that proposes an action never performs it. Proposing and committing are separate endpoints, and approval is the only path that mutates the live record — the propose path contains no write statements. (A demo reset endpoint reseeds the fixture; it's furniture I'd delete in production, and true structural separation — the propose path running without write permissions at all — is on the hardening list below.)

POST /api/copilot — propose, write nothing
Re-queries the live rows fresh (getTasks(), getWedding()), interpolates real data into every sentence, and drafts an action. The action is stashed server-side in a pendingActions Map with an id and a writes[] array. It is never executed on the way out. Returns { reply, action }; the client renders an action card.
POST /api/approve — the only write path
Looks the action up by id (an id the client didn't author). High-stakes actions are refused server-side unless confirmed === true. Then it runs the declared writes and nothing else: a complete_task is an UPDATE tasks SET status='done'; an update_guest_count adjusts the wedding row; every commit appends an audit_log entry. The pending action is deleted. Refresh and the change holds, it's a real node:sqlite row, not component state.

The tool surface

Writes are small typed operations, complete_task, update_guest_count, log_only, not one bundled plan_wedding. Today a regex picks the tool; a real LLM with tool-calling would pick the same tools. The pendingActions draft/commit split, the server-side gate, the audit trail, and the evals are all model-agnostic. The model is swappable behind the tools; the harness is the product.

Why this is the point, not a wedding toy

Swap the wedding's tasks table for a real domain ontology and the architecture is identical: atomic tools over a governed system of record, an approval gate, and an eval set that catches wrong actions. The wedding is a stand-in for the pattern, a governed record an agent can act on without drifting. I built the small version so the real one is a conversation, not a leap.

B2B2C data-product contract

The product is the governed record, not the chat surface.

For a wedding planning startup, architecture has to respect three actors: the planner buys and administers the workflow, the couple uses it in the messy middle, and vendors become referenced entities without getting full access. The copilot is only useful if the record is trustworthy enough to act on and constrained enough to audit.

  • Actor-aware recordEvery row needs an owner, freshness signal, and permission posture: planner-owned business state, couple-facing tasks, vendor references, and private notes should not collapse into one undifferentiated prompt context.
  • Action dictionary before model choiceTools declare read set, write set, reversibility, blast radius, and approval policy. A future LLM can pick tools, but it does not get to invent writes.
  • Commit-time validationDrafts should be cheap and reversible; commits should re-read live state, apply idempotently, and fail closed when the underlying record changed.
  • Audit as product UXPlanner trust comes from visible provenance: who asked, what was proposed, what was approved, what rows changed, and which eval/policy gate allowed it.
  • QA/evals as release policySeven binary evals are not demo tests; they are the release gate for model, prompt, and tool changes. Grounding, declared writes, high-stakes approval, handoff, judgment tone, and injection safety all have to stay green.
