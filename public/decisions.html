The engineering calls, and what each traded away

  • A deterministic rule engine plays the model on stage instead of a live LLM call in the demo. The model is commodity and swappable behind the same atomic tools. Keeping it deterministic forces the conversation onto the part that isn't commodity, the governed write path, not prompt quality. Every sentence is interpolated from live rows, so it's grounded by construction.
  • Split draft from commit, enforce the gate server-side instead of a UI-only confirm. A UI gate is a suggestion. Writes execute only inside /api/approve, keyed by a server-held id; replay a high-stakes approve with confirmed:false and the server rejects it, writes nothing. That's what eval 4 tests.
  • Atomic typed tools, not one bundled action instead of a single plan_wedding call. Small operations keep the blast radius declared and testable: approve flips exactly the declared rows and nothing else. It's also what makes the model swap-in trivial, the tool contract doesn't change.
  • Prototype in Lovable first; ship the demo as a stateful Node app instead of staying in the no-code mock. Lovable was the right first move — screens to react to in minutes — but the product’s core claim is a writeback that survives a refresh, and a mock holding component state can’t prove that. So the demo moved to a zero-dependency Node app with node:sqlite, deployed on Railway, where the hero beat is a real row flipping. The mock’s job was to be discarded.
  • Ground at request time; never cache state instead of holding a snapshot. copilot() re-queries every turn, so it's structurally grounded, not accurate-by-luck. It cannot name a vendor or date that isn't in the record. Fluency on data it can't see is the failure mode, not the feature.

The product calls, own one high-frequency job, cede generic inspiration, multiplayer by default, live in the pre-read. These are the calls under the hood.

Open questions & follow-ups

The design-review trail

The questions we push on internally, each with the current answer and the hardening step it points at. A prototype earns trust by naming its own seams.

  • Why is a rule engine playing the model? Isn’t that a fake demo? Deliberate. The model is the commodity layer — swappable behind the same three atomic tools — and keeping it deterministic keeps every design review on the part that isn’t commodity: the governed write path. Every sentence is interpolated from live rows, so grounding is by construction, not by luck. Next: Put a real model behind the tools (it only picks the tool; it still can’t write) and promote it only when it passes the eval contract on held-out cases.
  • What stops a replayed or forged approval? Writes execute only inside /api/approve, keyed by a server-held pending-action id the client didn’t author, and high-stakes actions are refused server-side unless confirmed === true. Replay a decline-cascade approve with confirmed:false and the server writes nothing — that’s eval 4. Next: Idempotency keys per approval, so even a legitimate double-submit can’t double-apply.
  • What if the record changed between draft and approve? Today the draft is grounded at propose time and the window is seconds long. The honest answer is that a stale draft could commit against moved state. Next: Re-validate live state at commit time, not just draft time, and carry the user’s edits to a draft through the approve payload.
  • What happens to a draft if the server restarts? It’s dropped — pendingActions is an in-memory Map. Acceptable for a prototype; drafts are cheap to regenerate. Next: Durable draft storage, so an approval in flight survives a restart.
  • How do the evals survive a real model? The contract survives; the assertions change. What counts as wrong — invented tasks, wrong vendor, undeclared writes, ungated cascades, false cheer, injection — is model-independent. The checks get rewritten from template assertions to behavior assertions. Next: The eval set becomes the release gate: a candidate model is promoted only if it passes without degrading held-out cases.
  • Two parties share this record. Where’s the access control? The planner view is read-only with the full audit trail, and every commit is attributed. But actor identity is a demo convention, not authentication. Next: Per-actor auth so the audit trail is trustworthy across parties, and PII in the record treated as an access-control boundary, not a column.
