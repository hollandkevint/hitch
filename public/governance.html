The write policy is the product

An agent earns autonomy by proving the conditions under which it's safe, not by demoing well. Four layers carry that here, and each maps directly onto what a real data product would need.

Write policy — value times reversibility

Cheap, reversible writes flow (reorder a to-do, mark a task done). Writes that change another party's reality get the gate. The gate names the cascade before it asks, guest count 120 → 118, the caterer's headcount, the seats, because a confirm that only says "are you sure" trains people to click through it. The policy is coded, not vibed.

The gate is server-enforced

There is no generic write endpoint to call. The only mutation path is /api/approve, and for high-stakes actions it checks confirmed === true and refuses server-side otherwise. The confirm dialog is not UI theater.

Audit trail and the counter-metric

Every commit writes an audit_log row — actor, action, timestamp — and the planner's read-only lens shows all of it. "Writebacks reversed" is the trust counter-metric to instrument first: if the agent starts acting wrongly, that number climbs and you see it. The trail makes every action attributable; the counter-metric makes trust measurable.

Eval set as the release gate

Seven binary checks run in one command — five specced before the build, two added from error analysis during it. Three are things a general model structurally cannot pass because it can't see the record. When there's a live model behind the tools, the eval contract becomes the regression gate: the assertions get rewritten from template checks to behavior checks, but what counts as wrong doesn't change, and a candidate model is promoted only if it passes without degrading the held-out cases. That's the difference between an agent that demos well and one that earns autonomy.

Where I'd harden it for production

Wrap multi-row writes in a transaction (the approve loop runs a couple of synchronous statements; I'd make that atomic). Add per-actor auth plus idempotency keys, so the audit trail is trustworthy across parties and a replayed approval can't double-apply. Re-validate live state at commit time, not just at draft time, and carry the user's edits to a draft through the approve payload. Move pendingActions from an in-memory Map to durable storage so a restart doesn't drop a draft mid-approval. Delete the demo reset endpoint, and run the propose path without write permissions at all. And treat PII in the record as a first-class access-control boundary, not a column. I know where the seams are because I built it, not spec'd it.

At scale

The same policy in a regulated domain

Swap the wedding’s tasks table for any governed domain record — a customer master, a clinical schedule, a territory model — and the questions don’t change: who may read which rows, which writes are cheap and which change another party’s reality, what proof gates a write, and how you measure whether trust is warranted.

The worked example above is the whole policy in miniature. “The Hendersons declined” is a two-guest RSVP change — but it moves the caterer’s headcount, contract exposure, and the seating math. Value times reversibility says that write gets a gate, and the gate names the cascade before it asks. The same test applied to a regulated record decides which agent actions flow, which get a human approval, and which the agent must refuse outright.

The instrumentation, not the vibes

Trust is measured, not asserted: the reversal counter-metric (writebacks the user reverses or edits) is instrumented first, and the PRD carries a kill test — reversals above ~15% after two iterations and writeback gets pulled while grounded answers stay. Governance that can’t lose an argument with its own data isn’t governance.
