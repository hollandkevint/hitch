How the PRD became a working product

The process story is the product judgment: start with the brief, get to a clean problem, build multiple artifacts, then make the demo prove the PRD instead of asking the room to imagine it.

  • Initial read and product framingRead the exercise PRD, wrote the high-level problem and decision basics, mapped the current state, then forced the first pass to name metrics, evidence quality, assumptions, and QA/eval gates.
  • Research and problem sketchingUsed manual write-ups, interview notes, wedding-planning research prompts, and a NotebookLM research notebook created through the CLI to stay close to user inputs, needs, and the planner/couple workflow. Sketched the problem space and architecture before polishing copy.
  • Prototype ladderStarted with quick artifacts and visual mocks, then moved through Claude Artifacts and Lovable-style prototype prompting into a real app shape. Used Claude Code and Compound Engineering passes to turn the Google Doc PRD into a working, inspectable build context.
  • Governed buildBuilt the questionnaire, record depth, planner view, approval gate, and audit trail in a Railway-deployed Node app with SQLite persistence. The architecture stayed centered on data governance: draft path writes nothing, commit path is server-gated, evals decide release.
  • Review hardeningRan code-review and doc-review passes, adversarial reads, and design polish. The output is not a slide: it is one live URL with the demo, PRD, assumptions, architecture, questions, and v2 preview ready for a panel working session.

The point is not that the prototype is perfect. The point is that the PRD, evals, architecture, and demo are the same product argument.
