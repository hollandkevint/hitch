Build first, evaluate from the build, then write the spec

I didn't write a long spec and hand it off. I wrote five evals before the product code, built the working thing, and evaluated from it — and error analysis during the build grew the set to seven. When the build is nearly free, the prototype becomes the specification, and the eval set becomes the definition of correct.

The sequence, in order: define what the agent must never do (five binary evals, written first) before a line of product code; build a real, running system, zero dependencies, so the mechanics are inspectable end to end; run the evals against the real server logic, not mocks, and let error analysis grow the set (a steady-counsel check and an injection-safety check earned their way in during the build); then write the 300-word PRD from a thing that already works. Prototyping-as-specification, not spec-then-prototype.

The eval set is the spec. I defined wrong before I built.

The very first screens were Lovable mocks — the right tool for reacting to a layout in minutes, the wrong tool for proving state — and the build deliberately left them behind (the decisions page carries that call). From there, tooling was Claude Code driving a zero-dependency Node app, node:http plus node:sqlite, deployed on Railway for real persistence. Zero deps was a deliberate constraint: every moving part, the tools, the gate, the audit trail, the evals, is legible in one file with nothing hidden behind a framework. The governance is the product, so the governance had to be readable.
