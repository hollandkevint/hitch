01 · Process

Build first, evaluate from the build, then write the spec

I didn't write a long spec and hand it off. I wrote the evals first, built the working thing, and evaluated from it. When the build is nearly free, the prototype becomes the specification, and the eval set becomes the definition of correct.

The sequence, in order: define what the agent must never do (the six binary evals) before a line of product code; build a real, running system, zero dependencies, so the mechanics are inspectable end to end; run the evals against the real server logic, not mocks; then write the 300-word PRD from a thing that already works. Prototyping-as-specification, not spec-then-prototype.

The eval set is the spec. I defined wrong before I built.

Tooling was Claude Code driving a zero-dependency Node app, node:http plus node:sqlite. Zero deps was a deliberate constraint: every moving part, the tools, the gate, the audit trail, the evals, is legible in one file with nothing hidden behind a framework. The governance is the product, so the governance had to be readable.

02 · Architecture

Draft and commit are separate paths. The write only fires through approval.

The whole safety story is one structural choice: the thing that proposes an action is incapable of performing it. Proposing and writing are different endpoints, and the model, or the rule engine standing in for it, never touches the write path.

POST /api/copilot — propose, write nothing
Re-queries the live rows fresh (getTasks(), getWedding()), interpolates real data into every sentence, and drafts an action. The action is stashed server-side in a pendingActions Map with an id and a writes[] array. It is never executed on the way out. Returns { reply, action }; the client renders an action card.
POST /api/approve — the only write path
Looks the action up by id (an id the client didn't author). High-stakes actions are refused server-side unless confirmed === true. Then it runs the declared writes and nothing else: a complete_task is an UPDATE tasks SET status='done'; an update_guest_count adjusts the wedding row; every commit appends an audit_log entry. The pending action is deleted. Refresh and the change holds, it's a real node:sqlite row, not component state.

The tool surface

Writes are small typed operations, complete_task, update_guest_count, log_only, not one bundled plan_wedding. Today a regex picks the tool; a real LLM with tool-calling would pick the same tools. The pendingActions draft/commit split, the server-side gate, the audit trail, and the evals are all model-agnostic. The model is swappable behind the tools; the harness is the product.

Why this is the point, not a wedding toy

Swap the wedding's tasks table for a real domain ontology and the architecture is identical: atomic tools over a governed system of record, an approval gate, and an eval set that catches wrong actions. The wedding is a stand-in for the pattern, a governed record an agent can act on without drifting. I built the small version so the real one is a conversation, not a leap.

03 · Decisions

The engineering calls, and what each traded away

  • A deterministic rule engine plays the model on stage instead of a live LLM call in the demo. The model is commodity and swappable behind the same atomic tools. Keeping it deterministic forces the conversation onto the part that isn't commodity, the governed write path, not prompt quality. Every sentence is interpolated from live rows, so it's grounded by construction.
  • Split draft from commit, enforce the gate server-side instead of a UI-only confirm. A UI gate is a suggestion. Writes execute only inside /api/approve, keyed by a server-held id; replay a high-stakes approve with confirmed:false and the server rejects it, writes nothing. That's what eval 4 tests.
  • Atomic typed tools, not one bundled action instead of a single plan_wedding call. Small operations keep the blast radius declared and testable: approve flips exactly the declared rows and nothing else. It's also what makes the model swap-in trivial, the tool contract doesn't change.
  • Ground at request time; never cache state instead of holding a snapshot. copilot() re-queries every turn, so it's structurally grounded, not accurate-by-luck. It cannot name a vendor or date that isn't in the record. Fluency on data it can't see is the failure mode, not the feature.

The product calls, own one high-frequency job, cede generic inspiration, multiplayer by default, live in the pre-read. These are the calls under the hood.

04 · Governance

The write policy is the product

An agent earns autonomy by proving the conditions under which it's safe, not by demoing well. Four layers carry that here, and each maps directly onto what a real data product would need.

Write policy — value times reversibility

Cheap, reversible writes flow (reorder a to-do, mark a task done). Writes that change another party's reality get the gate. The gate names the cascade before it asks, guest count 120 → 118, the caterer's headcount, the seats, because a confirm that only says "are you sure" trains people to click through it. The policy is coded, not vibed.

The gate is server-enforced

There is no generic write endpoint to call. The only mutation path is /api/approve, and for high-stakes actions it checks confirmed === true and refuses server-side otherwise. The confirm dialog is not UI theater.

Audit trail and the counter-metric

Every commit writes an audit_log row, actor, action, timestamp, and the planner's read-only lens surfaces "writebacks reversed." If the agent starts acting wrongly, that number climbs and you see it. Trust is measured, not asserted.

Eval set as the release gate

Six binary checks (seven with an injection-safety case) run in one command. Three are things a general model structurally cannot pass because it can't see the record. When there's a live model behind the tools, these same evals become the regression gate, promote a candidate only if it passes without degrading the held-out cases. That's the difference between an agent that demos well and one that earns autonomy.

Where I'd harden it for production

Wrap multi-row writes in a transaction (the approve loop runs a couple of synchronous statements; I'd make that atomic). Add per-actor auth so the audit trail is trustworthy across parties. Move pendingActions from an in-memory Map to durable storage so a restart doesn't drop a draft mid-approval. And treat PII in the record as a first-class access-control boundary, not a column. I know where the seams are because I built it, not spec'd it.

05 · Where it goes

Put a real model behind the tools

Going live changes almost nothing, and that's the point. The model's only new job is choosing which atomic tool to call; the tools, the gate, the audit trail, and the eval set stay exactly where they are. In agent terms the demo is a deterministic Category-1 agent; a real LLM makes it a Category-2 ReAct agent that decides sequence, but the tool surface and write policy are the same contract.

The open engineering question there isn't capability, it's economics: tool-selection is a bounded, cheap call, so you tier the model to the task, a premium model for the ambiguous judgment turns, a cheaper model for routine routing, routed through one gateway (pass-through pricing) so the swap is a config change. A copilot turn runs ~1.5K in / ~300 out, under a cent on a premium model like Sonnet 5 and roughly a sixth of that on an open model like GLM 5.2. At those numbers inference is a rounding error against the eval gate and the governed write path. The model is a commodity, and deflating; the record and the governance are the moat.
